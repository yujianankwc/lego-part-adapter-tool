# 零件适配中台 MVP PRD

## 1. 产品定位

零件适配中台用于将设计师提交的乐高 MOC 作品，快速转换为高砖可生产的零件清单（BOM）。

MVP 阶段目标不是做到 100% 自动替换，而是优先解决以下高频问题：

1. 乐高编号与高砖编号不同，实际同件但系统查不到。
2. 高砖有对应模具，但没有该颜色。
3. 高砖没有原件，但存在低风险相似替代件。
4. 印刷件、透明件、关键结构件需要单独标记与人工审核。
5. 把大批量逐件人工改稿，压缩为“系统自动替换 + 少量异常人工确认”。

MVP 成功标准：

1. 支持导入一份 BOM。
2. 自动完成精确替换、近色替换、部分高频相似件替换。
3. 输出可生产 BOM、替换报告、待审核清单。
4. 审核员可在系统内确认高风险项。

## 2. 目标用户

1. 运营同学：导入作品、跑适配、导出生产 BOM。
2. 审核同学：处理异常替换项，沉淀可复用规则。
3. 供应链同学：查看最终可生产零件清单与缺件清单。
4. 设计负责人：评估作品量产可行性、替换率、还原度风险。

## 3. MVP 范围

### 3.1 必做

1. BOM 导入（Excel/CSV 或粘贴文本）。
2. 零件编号精确映射。
3. 颜色近似替换。
4. 高频相似件替换（先覆盖基础砖、板、瓦、常见斜坡）。
5. 风险等级输出（A/B/C/D）。
6. 人工审核队列。
7. BOM/替换结果导出。

### 3.2 暂不做

1. 直接解析 Stud.io 三维模型。
2. 库存实时联动。
3. 自动成本优化。
4. 规则自动学习与推荐。
5. 多供应商零件库并行比价。

## 4. 核心流程

### 4.1 主流程

1. 用户导入 BOM。
2. 系统做字段标准化。
3. 适配引擎按规则顺序运行。
4. 系统输出自动适配结果与风险分级。
5. 审核员只处理 B/C/D 项。
6. 用户导出最终高砖 BOM 与替换报告。

### 4.2 规则执行顺序

1. 精确零件映射：乐高件 -> 高砖同模具件。
2. 精确颜色映射：若高砖有对应颜色，直接通过。
3. 近色替换：若颜色缺失，按保守/平衡/激进策略选近色。
4. 相似件替换：若模具缺失，按零件家族规则寻找替代。
5. 印刷件降级：优先同底色空白件，其次标记贴纸方案。
6. 无法处理：打入人工审核队列。

## 5. 页面设计

### 5.1 页面一：适配工作台

用途：完成导入、运行适配、查看整体结果。

关键区域：

1. 项目信息
2. BOM 导入区
3. 适配策略区
4. 结果 KPI 卡片
5. 自动替换结果表

字段建议：

1. 项目名称
2. 设计师名称
3. 来源文件
4. 颜色策略（保守/平衡/激进）
5. 是否允许展示件相似替换
6. 是否允许结构件相似替换

结果 KPI：

1. 总零件种类数
2. 总零件数量
3. 自动适配率
4. 待审核件数
5. 无法适配件数
6. 预计可生产率

### 5.2 页面二：异常审核台

用途：集中审核 B/C/D 级替换项。

列表字段：

1. 原零件编号
2. 原颜色
3. 数量
4. 系统建议方案
5. 风险等级
6. 风险原因
7. 审核动作（通过/改选/挂起）

批量能力：

1. 同一原件在同一项目里支持批量确认。
2. 审核结果可选择“保存为规则”。

### 5.3 页面三：规则管理（MVP 可先弱化）

用途：维护适配基础规则。

至少支持：

1. 零件精确映射表
2. 颜色映射表
3. 相似替换规则表
4. 禁止替换名单

MVP 可先作为后台静态配置，不必先做复杂编辑器。

## 6. 数据字段设计

### 6.1 项目导入结构

输入 BOM 标准字段：

1. `lego_part_no`
2. `lego_color_no`
3. `qty`
4. `part_name`（可选）
5. `is_printed`（可选）

### 6.2 结果结构

输出结果字段：

1. `original_part_no`
2. `original_color_no`
3. `resolved_part_no`
4. `resolved_color_no`
5. `qty`
6. `match_stage`（exact_part / exact_color / near_color / substitute / printed_fallback / manual）
7. `risk_level`（A / B / C / D）
8. `risk_reason`
9. `review_status`（auto_pass / pending_review / approved / rejected / hold）

## 7. 替换规则定义

### 7.1 风险等级

1. `A`：完全匹配，可直接通过。
2. `B`：可接受替换，建议复核。
3. `C`：高风险替换，必须人工确认。
4. `D`：无法可靠处理，必须人工改稿或特殊采购。

### 7.2 颜色策略

1. 保守：仅允许安全近色。
2. 平衡：允许安全近色 + 部分可接受近色。
3. 激进：允许更多近色，以提高可生产率。

硬规则：

1. 透明件只能替换为透明件。
2. 金属/珠光色尽量只在同表面效果内替换。
3. 肤色、角色主色、主题标志色默认禁止跨类替换。

### 7.3 相似件替换边界

优先覆盖：

1. 基础砖
2. 板件
3. 瓦片
4. 常见斜坡
5. 常见圆角件

暂缓覆盖：

1. 印刷件
2. 人仔相关
3. 稀有专用件
4. 透明特效件
5. 复杂机械件

## 8. 导出要求

MVP 至少输出 3 份结果：

1. 生产 BOM：给供应链直接使用。
2. 替换报告：记录每个零件如何被替换。
3. 异常清单：列出待审核与无法适配项目。

建议导出格式：

1. Excel（首选）
2. CSV（轻量）

## 9. 验收标准

### 9.1 功能验收

1. 可导入一份 50 行以上的 BOM。
2. 可完成自动替换并展示结果。
3. 可生成待审核列表。
4. 可人工修改审核状态。
5. 可导出结果文件。

### 9.2 业务验收

1. 首批高频规则可覆盖 60% 以上的“编号不同但同件”问题。
2. 加入颜色替换后，整体自动适配率达到 70% 以上。
3. 审核员仅需处理 10%-30% 的异常项。

## 10. MVP 开发阶段建议

### 第一阶段（本地可跑）

1. 用 mock 规则完成前端原型。
2. 支持手工粘贴 BOM。
3. 在页面里展示替换结果与审核队列。
4. 支持 CSV 导出。

### 第二阶段（接后端存储）

1. 接数据库保存项目与审核结果。
2. 接入规则管理接口。
3. 沉淀规则复用。

### 第三阶段（业务深化）

1. 接入库存数据。
2. 增加成本优先策略。
3. 支持多版本输出（高还原版 / 高量产版）。

## 11. 当前建议的实现方式

基于现有 `designer-plan-miniapp` 项目，优先在已有 FastAPI 后台中新增一个独立模块页：

1. 路径：`/admin/part-adapter`
2. 先使用静态页面 + 前端 mock 逻辑演示完整流程
3. 确认交互与规则方向后，再接入后端 API 与数据库

这样可以保持改动小、启动快、可立即在本地演示。
